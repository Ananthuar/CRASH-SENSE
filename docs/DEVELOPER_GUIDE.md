# CrashSense Developer Guide 📖

Welcome to the **CrashSense** codebase! This document serves as the absolute source of truth for understanding how the entire application works. If you are a new developer boarding the project, reading this guide will teach you exactly how the pieces fit together.

---

## 1. High-Level Architecture

CrashSense is an **Edge AI System Resolution Platform**. It is designed to run entirely locally on a bare-metal Linux machine to monitor processes, detect anomalies via Machine Learning, and proactively intervene before a system crash occurs.

The project is split into two massive tiers, connected by a standalone Flask REST API:

1. **The Backend Daemon (`backend/`)**: 
   A headless Python process that loops endlessly. It collects system data, feeds it to the AI, acts on the AI's conclusions using `sudo` signals, and serves that data over HTTP.
2. **The Desktop UI (`desktop/`)**: 
   A native Python GUI built entirely with `CustomTkinter`. It polls the backend API to render beautiful, hardware-accelerated dashboard and metrics interfaces.

---

## 2. The Backend Engine (The Brains)

The `backend/` folder contains the core logic of the application. It runs via `backend/app.py` leveraging a Flask application factory pattern.

### 2.1 The Polling Loop (`core/collector.py`)
Because we need absolute real-time analysis, `SystemMonitor.start()` spawns a daemon thread. It uses the `psutil` library to scan hardware (CPU, RAM, Disk I/O) every 500 milliseconds and stores exactly 120 historic snapshots in a rolling `deque`. 

### 2.2 The ML Model (`core/crash_predictor.py`)
Every few seconds, the backend runs the **Hybrid AI Crash Predictor**.
It is "Hybrid" because it ensembles two different Scikit-Learn approaches:
* **The Random Forest (Supervised)**: We pre-trained an ML model (`crash_rf_model.joblib`) on synthetic crash patterns (memory leaks, fork bombs, I/O storms) converted to normalized rates-of-change. It provides our baseline "Probability of Crash" (0-100%).
* **The Isolation Forest (Unsupervised)**: This algorithm fits itself live to the system's exact local footprint. It learns what "normal CPU/RAM noise" looks like for *this specific computer* and outputs an Anomaly Score.

### 2.3 Per-Process Monitoring (`core/process_monitor.py`)
Hardware metrics are too vague to kill a specific process. Therefore, `process_monitor.py` tracks every single POSIX process ID on the Linux machine simultaneously. 
It uses rigid algorithmic heuristics (not AI) to flag exactly *who* is causing the problem:
- **Memory Leak**: Monotonically growing RSS (>5MB/min for 2 mins).
- **CPU Runaway**: Sustained >90% CPU usage for 30s.
- **FD Exhaustion**: Nearing Linux file descriptor `ulimit`.

### 2.4 The Resolution Daemon (`core/resolution.py`)
When the `process_monitor` flags a "Critical" alert, the `SystemResolver` physically intervenes via standard Linux commands (using `sudo -n` for passwordless escalation):
- **Throttling**: Executes `renice 19 <pid>` on CPU hogs.
- **Termination**: Executes `kill -15 <pid>` on Memory Leaks or Zombie thrashing.
- **Cache Drops**: Executes `echo 3 > /proc/sys/vm/drop_caches` when the overall ML Model detects imminent OOM (Out-of-Memory) death.

## 3. The Desktop Application (The Face)

The `desktop/` directory contains the CustomTkinter GUI. It boots from `desktop/app.py`.

### 3.1 Routing & Navigation
We do not use messy multiple windows. `CrashSenseApp` acts as a Single-Page Application (SPA) router. All screens inherit from `CTkFrame` and are stored in the `self._screens` dictionary. Clicking the Sidebar simply calls `.pack_forget()` on the old screen and `.pack()` on the new one.

### 3.2 Firebase Auth (`firebase_service.py` & `desktop/auth.py`)
While everything runs locally, Firebase is used purely for Cloud Syncing (User Profiles and Settings). 
1. The Desktop UI opens a lightweight web browser window hitting the local Flask Auth endpoints.
2. The user logs in via Google/Email on the browser.
3. The Flask API receives the Auth token and writes the profile to Google Firestore (`firebase_service_account.json` required).

### 3.3 UI Rendering Highlights
* **Native OS Fallback**: If the app is minimized to the system tray (using `pystray`), crucial ML crash alerts bypass the GUI entirely and use Python's `subprocess` to trigger a Linux `notify-send` desktop banner.
* **Toasts**: In-app notifications (`desktop/components/notification_toast.py`) slide perfectly into view using recursive `.after()` commands, ensuring we never use `time.sleep()`, which would brutally freeze the tkinter main event loop.
* **Matplotlib integration**: The charts on the Dashboard are generated by Matplotlib and rendered into Tkinter canvases via `FigureCanvasTkAgg`.

## 4. Deployment Pipeline

CrashSense is not shipped as Python source code to clients. It is frozen into ELF binaries.

If you edit the code, you must package it using **`build.sh`**:
1. It uses **PyInstaller** to freeze the interpreter and the code.
2. It uses `--add-data` hooks to suck the Scikit-learn `.joblib` model and the `customtkinter` vector assets directly into the binary blob.
3. It creates an `install.sh` shell script.

When the end-user receives the resulting `/dist` folder on a totally offline, clean Linux VM, running `install.sh`:
- Moves the binaries to `/opt/crashsense`.
- Generates a `/etc/systemd/system/crashsense.service` payload so the backend engine survives reboots.
- Creates a Gnome `.desktop` graphical shortcut so the UI appears in their start menu.

*Note: Whenever resolving file paths (like loading `icon.png` or `model.joblib`), you will notice the code checks `try: sys._MEIPASS`. This is mandatory because frozen PyInstaller binaries extract to a random `/tmp` directory at runtime rather than `os.path.dirname(__file__)`.*
